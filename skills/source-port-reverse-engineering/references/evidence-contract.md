# Evidence Contract

Use this contract when designing tracked reverse-engineering evidence.

## Machine-Readable Reports

Every report should contain:

- schema version;
- source file names, sizes, and SHA-256 values;
- confirmed segmented/virtual/file addresses where applicable;
- static evidence and runtime evidence as separate fields;
- explicit bounded-path observation status;
- decoded record or command counts;
- terminal offset and trailing-byte count;
- unsupported or unexplained values as explicit failures;
- deterministic ordering and atomic publication.

Reject placeholders such as `unknown`, `hypothesis`, and `TBD` in reports that
claim completeness.

## Function And Global Catalogs

Track:

- address;
- descriptive name;
- status: `unknown`, `hypothesis`, or `confirmed`;
- concise evidence;
- intended native symbol when relevant.

Names are claims. Keep autogenerated names until evidence supports a semantic
name.

## Runtime Evidence

A reference run must define:

- exact executable and asset hashes;
- pinned emulator/tool version and hash;
- startup inputs and configuration;
- bounded start and stop conditions;
- observed calls, parameters, results, and caller addresses;
- cleanup in `finally`;
- original-asset verification before and after.

Record zero observed calls when absence on the bounded path matters.

## Format Recovery

For every format, establish:

- byte order and signedness;
- header and record layouts;
- dispatch values and payloads;
- loops, counts, terminators, and EOF rules;
- validation and error behavior;
- complete consumption of every accepted sample;
- worked examples tied to supplied bytes;
- loader/interpreter addresses.

Validate against all supplied same-format files accepted by the original
loader, not only the asset visible in one screenshot.

## Equivalence

Use deterministic logical inputs and exact outputs:

- indexed pixels and palette before RGB presentation where possible;
- audio/event streams before device output where possible;
- state snapshots and emitted actions for gameplay logic;
- exact comparison by default, with no tolerance added to hide mismatches.

An approved screenshot proves pixels and transitions only. It does not prove
the algorithm or resource format that produced them.
